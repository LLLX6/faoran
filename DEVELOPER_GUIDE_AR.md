# دليل تطوير خدماتي

هذا الدليل يوضح أماكن التعديل الآمنة في الإصدار الإنتاجي الحالي. الهدف أن يستطيع فريق الدعم متابعة المشروع من دون كسر توافق البيانات القديمة أو نسختي GitHub Pages والخادم.

## البنية المختصرة

| الجزء | الملف | المسؤولية |
|---|---|---|
| واجهة التطبيق | `index.html` | الحالة المحلية، الترجمة، التنقل، العرض، النماذج وربط API |
| تحسينات الإصدار | `assets/styles/khadamati-v54.css` | إصلاحات الهاتف وشاشة الخدمات والبطاقات والمحادثة |
| نسخة GitHub Pages | `public/index.html` و`public/assets/` | مرآة الواجهة الثابتة المنشورة |
| الخادم | `server.py` | الجلسات وAPI والتحقق والصور والتقارير والإشعارات |
| منطق النطاق | `khadamati_domain.py` | الاشتراكات والصلاحيات والمطابقة والموافقة والدفع |
| التخزين | SQLite / PostgreSQL | المصدر المشترك للحسابات والطلبات والاشتراكات |
| التثبيت والتحديث | `manifest.webmanifest` و`service-worker.js` | PWA والكاش والإشعارات |
| الاختبارات | `tests/` | رحلات الواجهة وAPI والأمان ومنطق النطاق والأداء |

## مصادر الحقيقة

1. الخادم هو مصدر الحقيقة للحسابات والصلاحيات والباقات والطلبات المشتركة والمحادثات. لا تمنح الواجهة صلاحية لا يتحقق منها الخادم.
2. `LocalStorage` يحفظ تفضيلات الجهاز ومسودات العمل وحالة العرض، ويعمل كطبقة توافق عند انقطاع الخادم. لا تستخدمه لحفظ أسرار أو كلمات مرور.
3. قواعد حدود الباقات والمطابقة موجودة في `khadamati_domain.py`. لا تكرر أرقام الخطط داخل شاشة جديدة.
4. قاموس الترجمة ودالة `L(ar,en)` هما المدخلان المعتمدان للنصوص. كل نص ظاهر يحتاج العربية والإنجليزية مع اختبار RTL/LTR.
5. `CONTACT_EMAIL` وإعداد `supportEmail` هما مدخل التواصل المركزي؛ لا تكتب بريدًا آخر يدويًا داخل الصفحات.

## خريطة الواجهة

- `renderEntry` و`renderProviderEntry`: بوابات البداية والدخول.
- `layout` و`render`: الغلاف العام، الرأس، شريط الهاتف واختيار الشاشة.
- `renderHome` و`renderServices` و`renderSearchV36`: رحلة اكتشاف الخدمة والمزود.
- `quickRequestForm` و`requestDraftFromForm`: إنشاء الطلب وموقعه وصوره.
- `requestBoardSheet`: ساحة الطلبات والترشيحات.
- `openRequestChat`: المحادثة والصور والصوت والموقع.
- `renderAccount` و`renderProviderApp` و`renderAdmin`: مساحات الحسابات الثلاث.
- `onClick`: موزع الإجراءات. أي إجراء جديد يجب أن ينتهي بـ `return` بعد معالجته حتى لا يصل إلى مسار توافق قديم.
- `normalizeProvider` و`normalizeRequestExtras`: قيم افتراضية للبيانات القديمة؛ حدّثهما عند إضافة حقل دائم.

## إضافة ميزة بأمان

1. افحص وجود وظيفة مشابهة قبل إنشاء دالة جديدة.
2. أضف قيمة افتراضية في طبقة التطبيع قبل قراءة الحقل الجديد.
3. أضف تحققًا في الواجهة لتحسين الرسالة، ثم تحققًا مماثلًا في الخادم للحماية.
4. أضف النص العربي والإنجليزي واختبر الاتجاهين.
5. أضف اختبارًا لمسار النجاح وحالة المنع أو الخطأ.
6. زامن `index.html` مع `public/index.html` وأي أصل جديد مع `public/assets/`.
7. حدّث رقم `APP_VERSION` واسم Cache في Service Worker عند تغيير ملفات الواجهة المنشورة.

## قواعد الصور والموقع

- الصور العامة تمر عبر أدوات القص والضغط الحالية قبل التخزين أو الرفع.
- الوثائق خاصة بالإدارة، ولا يجوز إدخال روابطها في بطاقات عامة.
- الموقع الدقيق لا يظهر إلا بعد إذن الجهاز وإجراء صريح من صاحبه.
- رسالة الموقع في المحادثة مرتبطة بطلب ومحادثة مقبولين، ولا تُرسل تلقائيًا.
- استخدم `normalizeCoordinates` في الواجهة و`normalized_location` في الخادم قبل أي حساب أو حفظ.

## الاختبارات قبل كل نشر

```powershell
python -m py_compile server.py khadamati_domain.py
python -m unittest discover -s tests -p "test_*.py" -v
python tests/security-api.py
node tests/smoke-ui.js
node tests/performance-ui.js
```

شغّل `tests/smoke-api.py` أمام خادم اختبار وقاعدة معزولة كما هو موضح في README. اختبر على الأقل 320 و390 و430 بكسل، ثم سطح المكتب، ولا ترفع إذا ظهر خطأ Console مؤثر أو فشل اختبار أساسي.

## ملاحظات صيانة

- ملف `index.html` كبير بسبب تراكم طبقات توافق لإصدارات وبيانات سابقة. لا تقسّمه قبل الإطلاق في تعديل واحد. نفّذ التقسيم لاحقًا على مراحل مع اختبارات تطابق DOM والبيانات بعد كل مرحلة.
- لا تعدّل قاعدة إنتاج مباشرة. أنشئ Backup ثم Migration قابلة للتكرار مرة واحدة.
- لا ترفع `.env` أو قاعدة البيانات أو مجلدات الرفع أو سجلات التشغيل.
- احتفظ بتطابق `index.html` و`public/index.html`، وتحقق منه آليًا قبل Commit.
